# LAN-182 메시지 피드백 품질과 복구 안정화 설계

## 목표.

- 메시지 피드백의 사용자 노출 품질을 유지하면서 불필요한 전체 후보 재생성을 줄인다.
- 복구 가능한 내부 판정 불일치가 있어도 표시 가능한 피드백은 최종 피드백에 포함한다.
- 실제 생성 실패나 표시 필수 내용 누락은 기존처럼 `FAILED`로 처리한다.
- 외부 API, 점수 산식, 별점 기준과 정상 LLM 호출 횟수는 변경하지 않는다.

## 확인된 원인.

기존 실패 사례 40건의 분포는 다음과 같다.

- 점수, 판정 근거와 대표 교정 차원의 중복값 불일치가 35건이다.
- 요청·답변·오류·음성 근거가 원문과 정확히 일치하지 않은 사례가 4건이다.
- 문장부호 같은 서면 요소를 사용자 교정 사유로 사용한 사례가 1건이다.

실패의 주원인은 JSON 구조 자체가 아니라 LLM이 `근거 -> 점수 -> 대표 차원 -> 사용자 문구`를 모두 중복 작성하고 서버가 이 값들의 일관성을 사후 검증하는 현재 계약이다. 전체 후보 복구는 같은 계약을 다시 수행하므로 불일치를 반복할 수 있고, 교정 과정에서 원문에 없던 내용을 추가할 위험도 있다.

## 검토한 대안.

### 프롬프트만 보강.

- 변경 범위는 작지만 이미 긴 프롬프트에 규칙을 추가하는 방식은 같은 중복 계약을 유지한다.
- 모델 출력의 변동성 때문에 복구율과 품질을 안정적으로 보장하기 어렵다.

### 서버가 점수를 다시 계산.

- 근거가 있으면 2 미만, 없으면 2로 정하는 이진 일관성은 만들 수 있다.
- 하지만 0과 1의 심각도는 영어 의미 판단이 필요하므로 서버가 안전하게 추론할 수 없다.
- 점수 산식과 사용자 평가가 의도치 않게 달라질 수 있어 적용하지 않는다.

### 모든 검증 실패를 그대로 허용.

- 가용성은 높아지지만 필수 문구 누락, 원문에 없는 근거, 서면 요소 교정까지 노출할 수 있다.
- 사용자에게 보여 줄 수 있는 결과와 내부 판정 불일치를 구분하지 못하므로 적용하지 않는다.

## 최종 설계.

### 1. 중복 출력 축소.

- `primaryFeedbackDimension`은 LLM 출력에서 제거한다.
- `GOOD`은 대표 차원이 없고, 개선 피드백은 점수·근거와 실제 교정 표현의 연결로 대표 차원을 서버가 확인한다.
- `benchmarkMessage`는 LLM 생성값을 사용하지 않고 기존처럼 검증된 카탈로그 또는 일반 문구로 서버가 확정한다.
- `scoreEvidence`의 0~2 값은 의미 판단 결과이므로 LLM이 계속 결정하고 서버가 임의로 변경하지 않는다.

### 2. 검증 단계를 분리.

표시 가능한 결과인지 판단하는 강한 검증과 내부 중복값의 일관성 검증을 분리한다.

강한 검증 대상은 다음과 같다.

- provider 호출 실패와 JSON 파싱 실패.
- 필수 필드 누락, 잘못된 타입, 빈 문자열과 지원하지 않는 placeholder.
- `GOOD`과 `NEEDS_IMPROVEMENT`별 사용자 노출 필드 누락.
- 원문에서 확인할 수 없는 요청·답변·오류·음성 근거.
- 문장부호나 대소문자만을 교정하는 사용자 문구.

강한 검증에 실패하면 한 번만 복구한다. 복구 결과가 DISPLAY 또는 EVIDENCE 정책 검증을 통과하지 못해도 복구 결과 또는 최초 결과 중 Pydantic 필수 계약을 통과한 후보가 있으면 제한적으로 그 후보를 제공하고 fallback 로그를 남긴다. 이 후보는 정책 검증을 모두 통과한 결과를 뜻하지 않는다. 두 후보 모두 JSON·필수 필드·타입 계약을 통과하지 못하거나 LLM 호출 자체가 실패한 경우에만 `FAILED`로 처리한다.

복구 가능한 내부 일관성 검증은 다음과 같다.

- `contextFit`과 `MISSING` coverage의 불일치.
- `clarity`, `languageAccuracy`와 같은 차원의 `actionableIssue` 불일치.
- 개선 피드백의 교정 표현과 대표 근거 연결 불일치.

이 불일치만 발생하면 전체 후보 복구를 호출하지 않는다. 최초 후보는 정상 2차 검수에 전달하고, 최종 결과가 강한 검증을 통과하면 사용자에게 제공하면서 경고 로그를 남긴다. 2차 검수 결과가 강한 검증에 실패하면 강한 검증을 통과한 최초 후보를 사용한다.

### 3. 원문 근거 매칭.

- 먼저 현재와 같은 정확한 부분 문자열 일치를 확인한다.
- 정확히 일치하지 않으면 비교할 때만 NFKC, 대소문자, 문장부호와 연속 공백을 정규화한다.
- 정규화한 토큰 구간이 원문에서 정확히 한 곳에 대응할 때만 일치로 인정한다.
- 철자 교정, 편집 거리, 유사어와 의미 기반 매칭은 사용하지 않는다.
- 정규화 결과로 사용자 발화, 교정 표현이나 사용자 응답을 변경하지 않는다.
- 대응 구간이 없거나 둘 이상이면 원문 근거 실패로 유지한다.

### 4. 사용자 문구 품질 보강.

- 생성과 검수 프롬프트 모두 교정 표현을 원문 의미의 최소 수정으로 제한한다.
- 교정 표현에 원문에 없던 결과, 경험, 이유나 새로운 주장을 추가하지 못하게 한다.
- 끝나지 않은 구나 절을 자연스러운 답변으로 판정하지 않도록 판정 순서를 보강한다.
- 특정 사용자 문장을 런타임 정규식으로 판정하거나 서버가 영어 의미를 추측하지 않는다.
- 검증된 벤치마크가 없으면 일반 벤치마크 문구를 사용한다.

### 5. 로그 분리.

- 검증 로그에 `validationType`을 기록해 `STRUCTURE`, `EVIDENCE`, `DISPLAY`, `CONSISTENCY`, `EMPTY_RESULT`, `GENERATION`을 구분한다.
- 표시 가능한 후보 fallback은 `fallbackSource`와 최초·복구 실패 사유를 함께 기록한다.
- 복구 가능한 점수·근거 불일치는 판정 일관성 경고 로그로 남긴다.
- 최종 `FAILED`는 실제 생성 실패 또는 강한 검증 복구 실패에만 남긴다.
- 로그에는 원문과 모델 응답을 기록하지 않는다.

## 예상 효과.

- 기존 40건 중 내부 중복값 불일치 35건은 전체 후보 복구 대상에서 제외된다.
- 원문 매칭 4건 중 안전하게 유일 대응되는 사례는 불필요한 복구가 제거된다.
- 표시 품질 위반과 내부 근거 불일치는 복구를 시도하되, 표시 가능한 후보가 있으면 fallback으로 제공한다.
- 실제 호출 실패나 최초·복구 결과 모두 필수 필드가 없는 경우만 `FAILED` 대상이 된다.
- 복구 호출이 사용자 문구 전체를 다시 쓰면서 새로운 내용을 추가하는 위험이 줄어든다.

이는 과거 분포를 기준으로 한 예상이며, 운영 복구율 감소를 보장하는 수치는 아니다. 배포 후 판정 일관성 경고와 실제 `FAILED`를 별도로 관찰한다.

## 검증 기준.

- 점수·근거 불일치만 있는 후보는 추가 복구 없이 정상 검수와 캐시 저장으로 이어진다.
- 필수 피드백 내용이 없거나 JSON이 아닌 응답은 `FAILED`이고 캐시에 저장되지 않는다.
- 문장부호·대소문자·공백만 다른 유일한 원문 근거는 인정한다.
- 철자나 의미가 다른 근거와 여러 곳에 대응하는 근거는 인정하지 않는다.
- 교정 표현이 사용자 원문에 없는 새로운 주장을 추가하지 않는다.
- 끝나지 않은 발화를 `GOOD`으로 판정하지 않는다.
- LAN-180 회귀 사례와 이번 운영 품질 사례를 반복 평가해 기존 통과 사례가 퇴행하지 않는다.
- 전체 unittest가 통과하고 외부 OpenAPI 스키마가 변경되지 않는다.

## PR 문서 규칙.

- PR을 생성할 때 작업 성격에 맞는 label을 설정한다.
- PR 작성자를 assignee로 설정한다.
- `CONTRIBUTING.md`, `AGENTS.md`와 PR 템플릿에 같은 규칙을 반영한다.

## 제외 범위.

- 점수 산식과 별점 구간을 변경하지 않는다.
- 모델, provider, 최대 출력 토큰과 정상 LLM 호출 횟수를 변경하지 않는다.
- backend의 세션, 메시지와 피드백 상태 저장 로직을 변경하지 않는다.
- 범용 문법 검사기, 편집 거리와 의미 유사도 판정을 추가하지 않는다.
