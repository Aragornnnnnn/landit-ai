# LAN-180 설계

## 목표.

- 명확한 문법·어휘·연어 오류가 있는 사용자 발화를 `GOOD`으로 과대 판정하지 않는다.
- 단순한 스타일 선호, 간결성 차이, 자연스러운 대안 표현은 교정 대상으로 만들지 않는다.
- 피드백 문구와 벤치마크가 실제 사용자 발화와 판정 근거에서 벗어나지 않도록 한다.
- 외부 `message-feedback` API 계약과 현재 2회 LLM 호출 구조는 유지한다.

## 확인된 문제.

- `contact number will be in your customer ID`처럼 의미·어휘상 교정이 필요한 표현이 `GOOD`으로 판정됐다.
- `luggage bag`, `another option about this situation`처럼 명확히 교정할 수 있는 어휘·연어 오류가 있는데도 전체 표현이 자연스럽다는 피드백이 생성됐다.
- 불규칙 과거형이나 3인칭 단수 표현이 없는 발화에 해당 패턴을 정확히 사용했다는 `benchmarkMessage`가 생성됐다.
- `correctionReason`은 한 가지 변경만 설명하지만 `correctionExpression`은 여러 문법과 표현을 함께 바꾸는 사례가 있었다.
- 교정 과정에서 `eat delivery food and beer`처럼 새로운 부자연스러운 표현이 만들어진 사례가 있었다.

최신 운영 데이터는 사용자 13명, 사용자 발화 50건이며 이 중 완료된 상세 피드백은 43건이다. 사용자 24번과 25번의 추가 데이터에서도 벤치마크 근거 불일치와 복수 교정 문제가 재현됐다.

## 판정 기준.

- `languageAccuracy`는 명확한 문법 오류뿐 아니라 학습자가 고쳐야 하는 명확한 어휘·연어 오류가 있으면 2보다 낮춘다.
- 의미가 통한다는 이유만으로 명확한 어휘·연어 오류를 `GOOD`으로 처리하지 않는다.
- 다른 표현이 더 흔하거나 간결하거나 관용적이라는 이유만으로는 점수를 낮추지 않는다.
- 필러, 즉시 반복, 자기 수정, 문장부호, 대소문자는 기존처럼 단독 교정 사유로 사용하지 않는다.
- 판단이 애매한 자연스러움 차이는 `GOOD`으로 유지하되, 실제 의미나 사용법이 잘못된 경우는 `NEEDS_IMPROVEMENT`로 판정한다.

## 내부 판정 계약.

- 후보 응답에 학습자에게 안내할 하나의 핵심 교정 차원인 `primaryFeedbackDimension`을 추가한다.
- 값은 `NONE`, `CONTEXT_FIT`, `CLARITY`, `LANGUAGE_ACCURACY` 중 하나로 제한한다.
- 모든 점수가 2인 `GOOD` 후보는 `NONE`만 허용한다.
- `NEEDS_IMPROVEMENT` 후보는 2보다 낮은 점수와 실제 근거가 있는 차원만 선택할 수 있다.
- `CONTEXT_FIT`을 선택하면 실제 `MISSING` coverage가 있어야 하고 교정 표현에는 구체적인 `[your ...]` placeholder가 있어야 한다.
- `CLARITY` 또는 `LANGUAGE_ACCURACY`를 선택하면 같은 차원의 `actionableIssue`가 있어야 하고 그 `correctionExcerpt`가 교정 표현에 포함되어야 한다.
- 점수 계산용 근거는 여러 문제를 포함할 수 있지만 학습자용 `positiveFeedback`, `correctionExpression`, `correctionReason`은 선택한 핵심 교정 하나에만 집중한다.

## 벤치마크 정책.

- 모델이 직접 작성한 `benchmarkMessage`는 검증된 패턴 근거 없이 그대로 노출하지 않는다.
- 카탈로그의 구체적인 벤치마크는 사용자 발화에 해당 패턴의 검증 가능한 정답 예문이 실제로 포함된 경우에만 사용한다.
- 패턴을 서버가 확실히 검증할 수 없으면 일반 벤치마크 문구를 사용한다.
- 정량 수치와 출처 문구는 기존처럼 검증된 카탈로그에서만 가져온다.

구체적인 벤치마크가 줄어들 수 있지만, 실제로 사용하지 않은 문법을 칭찬하는 것보다 일반 문구를 반환하는 편을 우선한다.

## 생성 및 검수 흐름.

1. 후보 생성 모델이 coverage, actionable issue, 점수, 핵심 교정 차원과 학습자용 문구를 생성한다.
2. 검수 모델이 원본 요청부터 근거를 다시 판정하고 핵심 교정 하나만 선택해 전체 후보를 교체한다.
3. 서버가 점수와 근거, 핵심 교정 차원, 교정 표현의 구조적 일관성을 검증한다.
4. 서버가 검증된 카탈로그 패턴 또는 일반 문구로 벤치마크를 확정한다.
5. 검증에 실패하면 기존 복구 및 `FAILED` 처리 흐름을 사용한다.

## 회귀 검증.

- 불규칙 과거형을 쓰지 않은 발화에는 불규칙 과거형 벤치마크가 붙지 않는다.
- 3인칭 단수 주어·동사를 쓰지 않은 발화에는 주어-동사 수 일치 벤치마크가 붙지 않는다.
- 명확한 어휘·연어 오류가 있는 발화는 `languageAccuracy` 근거를 갖춘 `NEEDS_IMPROVEMENT` 후보가 된다.
- 자연스러운 문법 대안, 필러, 즉시 반복만 있는 발화는 `GOOD`을 유지한다.
- `correctionReason`과 `correctionExpression`은 같은 핵심 교정 하나를 다룬다.
- 문맥 정보 누락 교정에는 구체적인 placeholder가 포함된다.
- 외부 OpenAPI 스키마는 변경되지 않는다.
- 전체 unittest와 실제 OpenRouter 회귀 표본 호출을 통과한다.

## 제외 범위.

- `next-message`, `inner-thought`, `closing-message`, `session-feedback` 품질은 변경하지 않는다.
- backend의 세션, 메시지, 피드백 처리 상태 저장 로직은 변경하지 않는다.
- LLM 호출 횟수를 늘리거나 새로운 모델·의존성을 추가하지 않는다.
