# LAN-167 메시지 피드백 생성·검수 재설계

## 배경과 근본 원인

LAN-167은 실제 사용자 데이터에서 피드백 품질을 높이기 위해 메시지 평가와 사용자용 문구 생성을 분리했다. 그러나 구현이 진행되면서 서버가 LLM의 의미 판단을 다시 해석하는 규칙 엔진으로 커졌다.

현재 흐름은 다음과 같다.

```text
LLM 판정
  -> 자유 형식 coreAsks와 evidence 생성
  -> 서버가 영문 키워드와 정규식으로 의미 재해석
  -> LLM 문구 생성
  -> 서버가 단어 겹침과 플레이스홀더 규칙으로 의미 재검증
```

이 구조에는 다음 근본 문제가 있다.

- `coreAsks`가 자유 문장인데 서버 후처리는 특정 영문 단어와 문장 패턴에 의존한다.
- 사용자 발화와 교정 표현의 의미 보존을 단어 겹침으로 판단해 동의어, 번역, 한국어 발화, 자연스러운 재구성을 안정적으로 처리하지 못한다.
- 하나의 가장 중요한 개선점을 제공해야 하는 제품 요구와 모든 미충족 요청의 근거 단어·플레이스홀더를 한 문장에 포함해야 하는 검증 규칙이 충돌한다.
- `languageAccuracy`가 낮으면 반드시 특정 문자열 교체가 있어야 한다는 규칙이 문법 오류 외의 뉘앙스, 공손함, 공격적 표현을 제대로 나타내지 못한다.
- 실제 실패를 고칠 때마다 사례별 정규식, 키워드, 템플릿과 테스트가 추가되어 다른 사례의 실패가 늘어나는 패치 순환이 발생한다.

현재 브랜치는 단순한 전체 피드백 생성·검수 기준점인 `44f51ff` 이후 10개 파일에서 4,706줄이 추가되고 878줄이 삭제됐다. 핵심 서비스에는 판정 재해석, 언어 교정 정규화, 누락 질문 추론, 허용 어휘 계산, 안전 템플릿 보정이 함께 들어가 있다.

고정 사례 21건은 모두 통과하지만 전체 115건의 저동시성 평가에서는 19건이 최종 실패했다. 주요 원인은 근거 단어 불일치 12건, 플레이스홀더 누락 3건, 기타 AI 응답 검증 실패 4건이었다. 실패 19건만 다시 실행해도 14건이 실패해 단순 모델 변동보다 검증 구조의 문제가 크다는 결론을 내린다.

## 목표

- 정상 경로의 모델 호출은 2회로 유지한다.
- 첫 번째 모델이 평가 근거와 완성된 피드백 초안을 함께 생성한다.
- 두 번째 모델이 원본 대화와 초안을 함께 검수하고 필요한 필드를 직접 다시 작성한다.
- 서버는 확실하게 판정할 수 있는 구조적 계약과 정량 규칙만 검증한다.
- 의미 보존과 피드백 유용성은 검수 모델과 오프라인 품질 평가로 관리한다.
- 사례별 정규식과 안전 템플릿을 추가하지 않고도 새 사용자 발화에 일반화한다.
- 기존 점수, 별점, `benchmarkMessage`, 캐시, 세션 피드백 동작을 유지한다.
- 외부 API 응답, OpenAPI 스키마, backend DTO와 DB 스키마를 변경하지 않는다.

## 비목표

- 서버에 영어 문법 검사기나 의미 비교 엔진을 구현하지 않는다.
- 특정 메시지 ID, 시나리오, 질문 문구에 맞춘 런타임 분기를 만들지 않는다.
- 모든 자연어 품질 오류를 런타임에서 결정적으로 차단하려 하지 않는다.
- 모델 호출을 한 번으로 줄이거나 모델 등급을 낮추는 최적화는 이번 범위에 포함하지 않는다.
- 세션 점수 정책과 검증된 패턴 catalog 정책을 다시 변경하지 않는다.

## 제안 구조

```text
원본 요청
  -> 1차 전체 피드백 생성
  -> 구조·정량 계약 검증
  -> 2차 전체 피드백 검수와 재작성
  -> 구조·정량 계약 검증
  -> benchmarkMessage catalog 후처리
  -> 캐시 저장과 세션 점수 계산
```

### 1차 전체 피드백 생성

첫 번째 호출은 내부 평가 근거와 사용자에게 보여 줄 전체 메시지 피드백을 한 번에 생성한다.

- `messageId`.
- `feedbackType`.
- `scoreEvidence.contextFit`.
- `scoreEvidence.clarity`.
- `scoreEvidence.languageAccuracy`.
- `baseLocaleAnalogy`.
- `positiveFeedback`.
- `feedbackDetail`.
- `correctionExpression`.
- `correctionReason`.
- `benchmarkMessage`.
- 내부 후처리용 `detectedPatterns`.

`scoreEvidence`는 지금처럼 각 항목에 0, 1, 2를 사용한다. 메시지 점수 계산식도 유지한다.

```text
messageScore = max(50, contextFit * 20 + clarity * 15 + languageAccuracy * 15)
```

세 항목이 모두 2이면 GOOD이며 하나라도 낮으면 NEEDS_IMPROVEMENT다. 이 관계와 계산 결과는 모델의 설명이 아니라 서버가 결정적으로 확인한다.

### 2차 전체 피드백 검수와 재작성

두 번째 호출은 원본 요청과 1차 전체 결과를 함께 받는다. 검수 모델은 승인 여부만 반환하지 않고 최종 전체 JSON을 반환한다. 문제가 없는 필드는 유지하고 문제가 있는 필드는 직접 다시 작성한다.

검수 모델은 다음 항목을 함께 확인한다.

- 질문과 시작 지시를 모두 고려한 `contextFit` 판정이 맞는지 확인한다.
- 의미 전달 가능성과 문법·어휘·뉘앙스·공손함 평가가 점수 근거에 맞는지 확인한다.
- 대문자, 마침표, 쉼표나 의미를 방해하지 않는 필러만으로 개선 피드백을 만들지 않는다.
- 문법적으로 모두 자연스러운 표현을 단순 선호 차이로 교정하지 않는다.
- 교정 표현이 사용자 발화의 의미, 의도, 시제, 부정 여부를 유지하는지 확인한다.
- 사용자 발화와 대화에 없는 이름, 지역, 취미, 감정, 습관, 경험, 이유를 추가하지 않는다.
- 필요한 정보가 없으면 `[your hobby]`, `[your hometown]`, `[your reason]`처럼 구체적인 플레이스홀더를 사용한다.
- 여러 내용을 묻는 질문에 일부만 답했다면 가장 중요한 누락 내용을 보완한다.
- 무관하거나 이해하기 어려운 발화를 문법적으로만 다듬지 않는다.
- 내부 생성 정책을 학습자 대상 문구에 노출하지 않는다.
- 다섯 사용자용 피드백 필드가 동일한 핵심 개선점을 설명하고 서로 모순되지 않게 한다.
- 근거 없는 형식적 칭찬을 만들지 않는다.

2차 결과는 최종 결과의 단일 소스다. 1차 결과와 필드별로 합치거나 1차 판정을 강제로 잠그지 않는다. 검수 과정에서 1차의 과대 판정이나 잘못된 교정 표현을 함께 고칠 수 있어야 한다.

## 서버 검증 경계

서버는 다음처럼 문자열만으로 확실하게 확인할 수 있는 계약만 검증한다.

- JSON 파싱과 Pydantic 스키마.
- 요청의 `messageId`와 응답의 `messageId` 일치.
- 필수 필드, 문자열 공백, enum과 nullability.
- `scoreEvidence` 각 값이 0, 1, 2인지 여부.
- `feedbackType`과 세 평가 항목의 GOOD/NEEDS_IMPROVEMENT 관계.
- GOOD과 NEEDS_IMPROVEMENT별 필드 계약.
- 플레이스홀더의 대괄호 문법.
- 사용자에게 노출하면 안 되는 내부 정책 문구.
- `detectedPatterns`의 catalog 등록 여부, 상태, gamifiable 값과 사용자 발화 내 정확한 evidence 존재 여부.
- 메시지 점수와 세션 점수의 정량 계산.

서버는 다음 의미 판단을 하지 않는다.

- 자유 문장 `coreAsks`의 영문 키워드 해석.
- 사용자 발화와 교정 표현 사이의 내용어 겹침 비율.
- 모든 미충족 질문에 대한 플레이스홀더 자동 추론.
- 자연스러운 대안 표현인지 판별하는 정규식.
- 특정 질문 조합을 위한 교정 표현 템플릿 선택.
- 단어 목록을 기반으로 한 사실 추가 여부 판단.
- 모델이 쓴 자연어 근거를 이용한 점수 강제 정규화.

이 경계를 넘는 품질 판단은 2차 검수 모델과 오프라인 회귀 평가가 담당한다. 런타임 검증을 통과했다는 사실은 자연어 품질이 완벽하다는 의미가 아니라 외부 계약을 안전하게 만족했다는 의미다.

## 실패 처리와 호출 수

- 정상 경로는 1차 생성 1회와 2차 검수 1회로 총 2회다.
- 1차 결과가 구조 검증에 실패하면 오류 원인과 후보를 사용해 1차 구조 복구를 한 번만 허용한다.
- 2차 결과가 구조 검증에 실패하면 오류 원인과 후보를 사용해 최종 구조 복구를 한 번만 허용한다.
- 복구 결과도 실패하면 요청을 실패시킨다.
- provider 또는 네트워크 오류에는 같은 요청 안에서 자동 재호출하지 않고 기존 503 계약을 유지한다.
- 검수되지 않은 1차 초안을 최종 fallback으로 저장하지 않는다.
- 의미 품질을 이유로 런타임 재호출하지 않는다. 의미 품질 실패는 오프라인 평가에서 발견하고 프롬프트나 모델 선택으로 해결한다.

정상 경로는 2회이며 한 단계가 구조 복구되면 3회, 두 단계가 각각 한 번 복구되면 최대 4회다. 복구는 예외 경로이며 반복 루프는 만들지 않는다.

## 유지할 정책

### benchmarkMessage

- `detectedPatterns`는 외부 응답에 노출하지 않고 AI 서버 내부 후처리에만 사용한다.
- GOOD이면서 catalog 등록, `status=correct`, `gamifiable=true`, 사용자 발화에 실제 evidence가 존재하는 패턴만 인정한다.
- 검증된 패턴이 있으면 catalog 문구로 `benchmarkMessage`를 덮어쓴다.
- 검증된 패턴이 없으면 검수 모델이 만든 안전한 비정량 문구를 사용한다.
- NEEDS_IMPROVEMENT의 `benchmarkMessage`는 `null`이다.

### 세션 점수와 별점

발화가 1개 또는 2개인 세션은 메시지 점수 평균을 반올림한다. 발화가 3개 이상이면 현재 적용된 원시 평균 70%와 GOOD 비율 30% 조합을 유지한다.

```text
numerator = sum(messageScores) * 7 + goodCount * 300
denominator = messageCount * 10
nativeScore = max(50, round_half_up(numerator / denominator))
```

별점은 최종 `nativeScore`로 계산하는 기존 매핑을 유지한다.

- 0~54점은 1.0개다.
- 55~64점은 1.5개다.
- 65~74점은 2.0개다.
- 75~89점은 2.5개다.
- 90~100점은 3.0개다.

## 제거와 단순화 범위

다음 내부 모델과 이를 지원하는 런타임 의미 규칙은 제거 대상으로 삼는다.

- `MessageFeedbackJudgement`.
- `MessageFeedbackCopy`.
- `MessageFeedbackCoreAsk`.
- `MessageFeedbackLanguageCorrection`.
- `coreAsks`, `statedFacts`, `languageCorrections`, `requiredPlaceholder` 기반 후처리.
- `_meaningful_evidence_words`를 포함한 내용어 허용 목록 검증.
- 질문 키워드에 따른 플레이스홀더 추론.
- 사례별 안전 교정 템플릿과 일반 평가 정규화.
- `message_feedback_copy_unsupported_content`처럼 어휘 추정에 의존하는 실패 코드.

단, 제거 전 각 로직이 외부 계약, benchmark catalog 검증, 점수 계산에 실제로 사용되는지 호출 경로를 확인한다. 이번 변경이 만든 고립 코드와 테스트만 제거하고 LAN-166의 점수 정책과 benchmark catalog 로직은 보존한다.

테스트는 메시지별 사례를 런타임 정규식에 고정하는 방식에서 다음 계약과 품질 경계 중심으로 재구성한다.

- 정상 경로가 정확히 2회 호출되는지 확인한다.
- 2차 결과가 1차 결과의 판정과 문구를 모두 수정할 수 있는지 확인한다.
- 구조 오류가 한 번만 복구되고 반복되지 않는지 확인한다.
- GOOD/NEEDS_IMPROVEMENT 필드 계약과 점수 계산을 확인한다.
- 플레이스홀더 문법과 내부 정책 문구 차단을 확인한다.
- benchmark catalog와 외부 OpenAPI 비노출 계약을 확인한다.
- 대표적인 품질 사례는 실제 네트워크 없는 회귀 fixture로 유지한다.
- 실제 모델의 일반화 품질은 별도 평가 스크립트로 확인한다.

## 운영 검증 기준

코드 변경은 다음 순서로 검증한다.

1. 관련 단위 테스트와 API 테스트를 통과한다.
2. `.venv/bin/python -m unittest discover -s tests`를 통과한다.
3. `.venv/bin/python -m compileall app tests scripts`를 통과한다.
4. `.venv/bin/python -m pip check`를 통과한다.
5. FastAPI OpenAPI 스키마에서 외부 응답 필드가 변경되지 않았음을 확인한다.
6. 고정 품질 사례 21건을 모두 통과한다.
7. 중요 사례 42건에서 최종 구조 실패와 중대한 사실 추가가 0건인지 확인한다.
8. 전체 115건에서 최종 구조 실패가 최대 1건인지 확인한다.
9. 전체 데이터의 GOOD/NEEDS_IMPROVEMENT, 점수, 사용자용 다섯 필드를 변경 전후로 표본 검토한다.

전체 115건 평가는 단순 성공률만 보지 않는다. 다음 중대 품질 오류도 함께 집계한다.

- 질문의 핵심 요구를 놓친 GOOD 과대 판정.
- 사용자에게 없는 구체 사실 추가.
- 의미, 의도, 시제나 부정 여부 변경.
- 문장부호, 대문자, 필러나 단순 선호 차이만을 개선점으로 제시.
- 질문과 무관한 교정 표현.
- 피드백 필드 간 핵심 개선점 불일치.

운영 기준을 충족하지 못하면 새 정규식이나 사례별 템플릿을 추가하지 않는다. 실패를 생성 프롬프트, 검수 프롬프트, 모델 역량, 구조 계약 중 하나로 분류하고 해당 계층만 수정한다. 두 번의 원인 기반 수정과 전체 재평가 뒤에도 기준을 충족하지 못하면 정상 경로 호출 수나 외부 계약을 임의로 바꾸지 않고 별도 결정을 요청한다.

## 예상 효과와 위험

예상 효과는 런타임 코드와 테스트를 크게 줄이면서 새 질문과 발화에 대한 일반화 가능성을 높이는 것이다. 자연어 의미 검증을 서버 정규식에서 제거하므로 거짓 거부와 패치 연쇄도 줄어든다.

남는 위험은 2차 검수도 확률적인 모델 호출이라는 점이다. 이를 서버의 불완전한 의미 규칙으로 감추지 않고 고정 fixture, 전체 115건 평가, 중대 오류 집계와 모델 선택 기준으로 관리한다. 구조 안전성은 서버가 보장하고 자연어 품질은 모델과 오프라인 품질 게이트가 보장하는 경계를 유지한다.
