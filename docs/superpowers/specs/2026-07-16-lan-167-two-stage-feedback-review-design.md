# LAN-167 메시지 피드백 2단계 검수 설계

## 배경

LAN-167의 단일 호출 프롬프트에는 다중 질문 충족, 사용자 사실 보존, 플레이스홀더 사용, 대문자와 문장부호만을 근거로 한 교정 금지 규칙이 이미 포함돼 있다. 실제 사용자 발화 115개를 평가한 결과, GOOD 22개 중 9개가 복합 질문을 일부만 답했는데도 GOOD으로 판정됐고 NEEDS_IMPROVEMENT 93개 중 55개에서 교정 품질 문제가 발견됐다. 중요 사례 14개를 세 번 평가했을 때 9개는 세 번 모두 문제가 재현돼 프롬프트 문구만 추가하는 방식으로는 품질을 안정화하기 어렵다.

## 목표

- 첫 번째 모델 호출이 만든 메시지별 피드백을 두 번째 모델 호출이 원본 평가 문맥과 비교해 검수하고 필요하면 수정한다.
- 복합 질문 누락, 질문과 무관한 교정, 사용자 발화에 없는 의미 추가, 대문자·문장부호만을 근거로 한 교정을 우선적으로 막는다.
- 외부 메시지 피드백 API, backend DTO, DB 스키마는 변경하지 않는다.
- 두 번째 호출 실패가 마지막 턴 전체 실패로 이어지지 않도록 검증된 첫 번째 결과를 fallback으로 사용한다.

## 선택한 접근

모든 메시지 피드백에 동일 모델을 순차로 두 번 호출한다.

1. 생성 단계는 현재 메시지 피드백 프롬프트로 후보 피드백과 `scoreEvidence`, `detectedPatterns`를 생성한다.
2. 검수 단계는 원본 시나리오, 평가 문맥, 사용자 발화와 생성 단계 후보를 함께 받아 최종 피드백 전체를 다시 반환한다.
3. 검수 단계는 후보를 그대로 승인하는 별도 상태값을 만들지 않는다. 변경이 없더라도 외부 계약과 동일한 완성된 JSON을 반환한다.
4. 최종 후보는 기존 Pydantic 모델과 메시지 ID 검증을 통과해야 한다.
5. 검수 호출 실패나 검수 결과 검증 실패 시 생성 단계에서 이미 검증된 후보를 사용한다.

이 방식은 단일 호출 프롬프트 강화보다 비용과 지연이 크지만, 동일 모델이 생성 작업과 검수 작업을 분리해 수행하므로 반복 재현된 의미 오류를 직접 다시 확인할 수 있다. 규칙 기반 문자열 검사는 복합 질문 충족이나 의미 보존을 일반적으로 판단할 수 없어 선택하지 않는다.

## 검수 규칙

검수 프롬프트는 다음 순서로 후보를 확인한다.

1. 평가 문맥의 독립적인 핵심 요청을 모두 식별한다.
2. 사용자 발화가 각 핵심 요청을 충족했는지 확인하고, 하나라도 빠지면 `contextFit=1` 이하와 `NEEDS_IMPROVEMENT`로 수정한다.
3. `correctionExpression`이 평가 문맥에 직접 답하는지 확인한다. 무관한 사용자 문장을 자연스럽게 다듬는 데 그치면 문맥에 맞는 답으로 교체한다.
4. 교정 표현의 이름, 장소, 취미, 감정, 습관, 선호, 경험, 이유가 사용자 발화·평가 문맥·시나리오에 근거하는지 확인한다. 근거가 필요한 개인 정보는 구체적인 대괄호 플레이스홀더로 바꾼다.
5. 사용자 발화의 의도, 시제, 부정과 이미 나온 사실을 유지한다.
6. 의미를 바꾸지 않는 대문자, 문장부호, 필러, 자연스러운 문법 대안만 차이가 난다면 GOOD으로 수정한다.
7. `correctionReason`은 학습자가 실제로 보완할 내용만 설명하며 내부 생성 정책을 노출하지 않는다.

## 데이터 흐름

- 생성 단계의 원시 응답에서 `detectedPatterns`를 분리한 뒤 `MessageFeedbackEvaluation`로 검증한다.
- 검수 단계에는 원본 요청과 검증된 생성 후보를 전달한다.
- 검수 응답도 `detectedPatterns`를 분리한 뒤 같은 모델로 검증한다.
- 검수 응답이 유효하면 그 결과와 검수 응답의 `detectedPatterns`를 최종값으로 사용한다.
- 검수 단계가 실패하면 생성 후보와 생성 응답의 `detectedPatterns`를 최종값으로 사용한다.
- 최종값에만 기존 catalog 기반 `benchmarkMessage` 후처리를 적용하고 캐시에 저장한다.
- 생성 후보와 검수용 내부 데이터는 외부 API 응답과 세션 피드백 응답에 노출하지 않는다.

## 실패 처리

- 생성 단계의 호출·JSON 파싱·DTO 검증이 실패하면 기존과 동일하게 요청을 실패시킨다.
- 생성 단계가 유효한 뒤 검수 단계의 호출, JSON 파싱, DTO 검증 또는 메시지 ID 검증이 실패하면 생성 후보로 fallback한다.
- fallback을 위해 추가 재시도 호출은 하지 않는다. 정상 경로는 두 번, 실패 경로도 최대 두 번의 모델 호출로 제한한다.
- fallback 여부는 응답 계약에 새 필드를 추가하지 않는다. 기존 비민감 메타데이터 로깅 범위 안에서 추적할 수 있도록 검수 실패 로그만 남긴다.

## 테스트

- 두 번의 모델 응답을 순서대로 반환하는 테스트 대역으로 생성 후보가 검수 결과로 교체되는지 확인한다.
- 복합 질문을 일부만 답한 후보 GOOD이 NEEDS_IMPROVEMENT로 수정되는지 확인한다.
- 질문과 무관한 교정이 평가 문맥에 맞고 플레이스홀더를 사용하는 표현으로 수정되는지 확인한다.
- 사용자에게 없는 이유·이름·경험을 추가한 후보가 사실 보존 표현으로 수정되는지 확인한다.
- 대문자·문장부호만 고친 NEEDS_IMPROVEMENT가 GOOD으로 수정되는지 확인한다.
- 검수 호출 실패, 잘못된 JSON, DTO 불일치, 메시지 ID 불일치에서 생성 후보로 fallback하는지 확인한다.
- 기존 benchmark catalog 후처리와 외부 OpenAPI 스키마가 유지되는지 확인한다.
- 전체 unittest 후 실제 사용자 데이터의 중요 사례 14개를 여러 번 평가해 반복 실패 수를 비교한다.

## 성공 기준

- 외부 API와 backend 변경 없이 2단계 검수 결과가 캐시에 저장된다.
- 검수 실패 시 유효한 생성 후보가 정상 저장된다.
- 중요 사례 14개를 3회씩 평가했을 때 동일한 중대 문제가 세 번 모두 재현되는 사례가 기존 9개보다 감소한다.
- `.venv/bin/python -m unittest discover -s tests`가 통과한다.
