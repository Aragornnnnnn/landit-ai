# LAN-175 2회 피드백 재심사와 실제 세션 품질 검증 설계

## 배경

전달받은 세션 18에서는 세 사용자 발화가 모두 `NEEDS_IMPROVEMENT`로 저장됐다.

- 이름과 출신을 말한 자기소개에 모델이 말하지 않은 취미를 요구했다.
- 음성 발화의 `I I` 반복만을 교정 대상으로 삼았다.
- 복수 주어에 `is`를 사용한 실제 오류는 찾았지만, 교정 이유를 `be동사를 넣어서`라고 잘못 설명했다.
- 세 메시지가 모두 개선 필요로 저장됐는데도 세션 총평은 `모두 자연스럽게`라고 칭찬해 메시지 피드백과 모순됐다.

현재 LAN-167 구조는 첫 번째 호출의 `scoreEvidence`와 `feedbackType`을 서버에서 잠그고, 두 번째 호출은 사용자용 문구만 작성한다. 이 구조는 문구가 판정을 뒤집는 것을 막지만 첫 번째 호출의 의미 오판도 그대로 고정한다. LAN-175의 임시 구현은 자기소개 문구, 반복 단어와 교정 이유를 정규식과 단어 목록으로 탐지해 점수를 바꾸므로 새로운 질문과 표현에 일반화되지 않는다.

## 목표

- 정상 경로의 메시지 피드백 LLM 호출을 2회로 유지한다.
- 두 번째 호출이 원본 요청과 첫 번째 후보를 독립적으로 재검토하고 최종 점수, 유형과 문구를 함께 확정하게 한다.
- 특정 질문, 사용자 문장, 메시지 ID 또는 한국어 교정 문구에 의존하는 의미 판정 하드코딩을 제거한다.
- 구조와 음성 서비스에서 결정적으로 확인할 수 있는 서버 검증은 유지한다.
- 전달받은 세션 행의 AI 질문과 사용자 발화를 수정하지 않은 fixture로 저장하고 OpenRouter에서 반복 재현한다.
- 메시지별 결과와 세션 총평, 점수, 별점을 한 보고서에서 확인한다.
- 기존 외부 API, OpenAPI, 백엔드 DTO와 DB 스키마를 유지한다.

## 비목표

- 범용 영어 문법 검사기나 규칙 엔진을 서버에 구현하지 않는다.
- LLM 호출을 3회로 늘려 다수결이나 별도 문구 생성 단계를 추가하지 않는다.
- 모델, 최대 출력 토큰과 점수 산식을 변경하지 않는다.
- provider 또는 네트워크 실패에 임의 점수나 일반 피드백을 생성하지 않는다.
- LAN-175와 무관한 대화 생성, 속마음 또는 마무리 메시지 동작을 변경하지 않는다.

## 제안 구조

```text
원본 요청
  -> 1차 전체 후보 생성
  -> 서버의 구조 검증
  -> 2차 전체 재심사와 최종 문구 생성
  -> 서버의 구조·스피킹 검증
  -> 성공 시 2차 점수와 문구 저장
  -> 실패 시 검증된 1차 후보 저장
  -> benchmarkMessage catalog 후처리
  -> 캐시 저장과 세션 점수 계산
```

## 1차 전체 후보

첫 번째 호출은 현재처럼 다음 값을 포함한 fallback 가능한 전체 후보를 만든다.

- `scoreEvidence.contextFit`.
- `scoreEvidence.clarity`.
- `scoreEvidence.languageAccuracy`.
- `baseLocaleAnalogy`.
- `positiveFeedback`.
- `feedbackDetail`.
- `correctionExpression`.
- `correctionReason`.
- `benchmarkMessage`.
- 내부 catalog 후처리용 `detectedPatterns`.

`messageId`는 요청값을 사용하고 `feedbackType`은 서버가 세 점수에서 계산한다. 첫 번째 결과가 구조 검증을 통과해야만 두 번째 호출 실패 시 사용할 수 있는 fallback 후보가 된다.

## 2차 전체 재심사

두 번째 호출은 문구만 다듬지 않고 원본 요청과 1차 후보를 함께 검토한다. 출력은 1차와 같은 `scoreEvidence`, 사용자용 문구와 `detectedPatterns`를 가진다. 서버는 2차 `scoreEvidence`에서 최종 `feedbackType`을 다시 계산한다.

재심사는 다음 순서로 판단한다.

1. 현재 AI 질문에서 독립적으로 요구한 항목을 식별한다.
2. 사용자 발화가 각 항목에 답했는지 확인한다.
3. 음성 입력의 필러, 즉시 반복과 자기수정을 의미 또는 문법 오류와 구분한다.
4. 반복을 제거하고 남은 발화에 실제 문법, 명료성 또는 문맥 오류가 있는지 확인한다.
5. 점수와 유형에 맞는 사용자용 문구를 작성한다.
6. `correctionExpression`과 `correctionReason`이 같은 실제 변경을 설명하는지 자체 검증한다.

재심사 프롬프트에는 특정 세션 문장을 정답으로 넣지 않고 일반 원칙과 대비 예시를 사용한다.

- 열린 자기소개 질문은 이름과 하나 이상의 구체적인 개인 정보로 답할 수 있으며 취미를 반드시 요구하지 않는다.
- `I I`나 `is is`처럼 즉시 반복된 음성 토큰은 한 번으로 읽고, 반복을 제거한 문장에 남은 오류만 평가한다.
- 여러 장소를 나열한 복수 주어에 단수 동사를 쓴 경우에는 반복과 별개로 주어와 동사의 수 일치를 평가한다.
- 교정 표현이 `is`를 `are`로 바꾸면 교정 이유도 복수 주어와 동사 일치를 설명해야 한다.

## 서버 검증 경계

서버는 의미를 다시 추측하지 않고 다음 결정적 계약만 검증한다.

- 요청의 `messageId` 주입.
- `scoreEvidence`에서 `feedbackType` 계산.
- 유형별 필수 필드와 `null` 필드.
- Pydantic 스키마와 빈 문자열 거부.
- `[your ...]` 플레이스홀더 형식.
- 사용자 발화와 대소문자, 문장부호, 공백만 다른 교정을 스피킹 개선으로 거부.
- 사용자용 문구가 대소문자나 문장부호만을 개선 이유로 제시하는 경우 거부.
- catalog에 없는 정량 `benchmarkMessage` 거부.

다음 LAN-175 임시 의미 판정은 제거한다.

- 질문에 `name`, `about yourself`가 있는지 검사하는 자기소개 정규식.
- 사용자 발화에서 `from`을 찾는 정규식.
- 반복 단어와 한국어 `반복`, `중복`, `하나만`을 결합한 점수 변경.
- `복수 주어`, `be동사`, `are` 같은 교정 이유 단어 목록으로 오류를 재판정하는 분기.
- 세션 총평의 특정 한국어 칭찬 목록을 치환하는 후처리.

## 실패 처리와 호출 횟수

- 정상 메시지 피드백은 1차 후보와 2차 재심사로 총 2회 호출한다.
- 각 단계가 JSON 구조 검증에 실패하면 해당 단계만 한 번 복구할 수 있다.
- 2차 provider 오류 또는 구조 복구 실패 시 검증된 1차 후보를 저장한다.
- 1차 provider 오류 또는 1차 구조 복구 실패 시 임의 피드백을 저장하지 않고 기존 실패 계약을 유지한다.
- 추가 blind retry와 3차 의미 판정은 도입하지 않는다.
- 내부 cache와 품질 보고서에는 후보 복구, 재심사 복구와 fallback 여부를 기록하되 외부 API에는 노출하지 않는다.

## 세션 총평 일관성

세션 총평은 캐시된 최종 메시지 피드백을 입력으로 사용한다. 프롬프트에 다음 자체 검증을 추가한다.

- `NEEDS_IMPROVEMENT`가 하나라도 있으면 모든 답변이 자연스럽거나 완벽했다는 절대 표현을 사용하지 않는다.
- 강점은 실제 `GOOD` 피드백 또는 각 `NEEDS_IMPROVEMENT`의 `positiveFeedback`에서만 가져온다.
- 개선 방향은 실제 교정 이유와 모순되지 않게 한 가지로 요약한다.

서버에서 한국어 칭찬 단어 목록을 찾아 고정 문구로 교체하지 않는다. 세션 점수와 별점은 최종 재심사 점수로 기존 산식을 그대로 적용한다.

## 전달 데이터 재현 fixture

품질 fixture에는 전달받은 세션 18의 다음 원문을 그대로 저장한다.

- 세 AI 질문 59, 61, 63의 `message_content`.
- 세 사용자 발화 60, 62, 64의 `message_content`.
- 시나리오 제목, 목표와 상대 역할.
- 세션 피드백 요청에 필요한 메시지 ID와 순서.

오탈자, 필러, 반복, 문장부호와 지명 표기를 평가 전에 정리하지 않는다. 특히 `I I`, `is is`, `city city`, `Kwangwala Beach`, `I just yeah. Came here.`를 원문 그대로 유지한다. 기존 저장 피드백과 세션 총평은 회귀 원인 자료로만 보며 모델 입력이나 정답 문구로 사용하지 않는다.

기대 경계는 다음과 같다.

| 메시지 | 기대 유형 | 기대 점수 근거 | 기대 메시지 점수 |
| --- | --- | --- | --- |
| 60 자기소개 | `GOOD` | `2, 2, 2` | 100 |
| 62 취미와 이유 | `GOOD` | `2, 2, 2` | 100 |
| 64 여행지 추천 | `NEEDS_IMPROVEMENT` | `contextFit=2`, `clarity=1~2`, `languageAccuracy=1` | 70~85 |

메시지 64의 원문은 질문에 필요한 장소와 이유를 모두 전달하지만 문장 단편과 여러 자기수정이 있어 `clarity` 1과 2가 모두 합리적인 경계다. 이를 반영해 세 메시지의 기존 세션 산식을 적용한 기대 `nativeScore`는 83~87이고 `starRating`은 2.5다. 세션 총평의 정확한 문장은 고정하지 않지만 다음은 허용하지 않는다.

- 모든 답변이 자연스럽거나 완벽했다는 주장.
- 사용자가 말하지 않은 취미, 이유, 경험 또는 장소 추가.
- 실제 개선점과 무관한 일반 문법 조언.

## 자동 테스트

단위 테스트는 다음 동작을 RED부터 고정한다.

- 2차 재심사가 1차 `NEEDS_IMPROVEMENT`를 `GOOD`으로 바꾸고 최종 점수 근거가 캐시에 저장된다.
- 2차 재심사가 1차 `GOOD`을 `NEEDS_IMPROVEMENT`로 바꿀 수 있다.
- 2차 출력의 `feedbackType`은 받지 않고 서버가 2차 점수에서 계산한다.
- 2차 재심사 provider 오류와 구조 복구 실패에서는 검증된 1차 후보와 1차 점수를 저장한다.
- 자기소개, 음성 반복과 복수 주어를 판정하는 런타임 정규식과 단어 목록이 없다.
- 세션 총평 프롬프트가 혼합 피드백에서 절대 칭찬을 금지한다.
- 전달 fixture의 AI 질문과 사용자 발화가 첨부 원문과 정확히 일치한다.
- 품질 평가 도구가 한 세션의 메시지별 최종 결과, 세션 총평, `nativeScore`, `starRating`, 복구와 fallback 여부를 기록한다.
- 외부 응답과 OpenAPI에 `scoreEvidence`, `detectedPatterns`와 내부 품질 지표가 노출되지 않는다.

## OpenRouter 실제 품질 검증

구현과 전체 로컬 검증이 끝난 뒤 실제 설정의 `openai/gpt-5.4-mini`로 전달 세션을 3회 재생한다. 각 재생은 세 메시지 피드백을 순서대로 만든 뒤 같은 cache에서 세션 피드백까지 생성한다.

성공 기준은 다음과 같다.

- 3회 모두 메시지 60과 62가 `GOOD`, 메시지 64가 `NEEDS_IMPROVEMENT`다.
- 3회 모두 점수 근거와 메시지 점수가 기대 경계와 일치한다.
- 메시지 60에 취미 누락 피드백이 없다.
- 메시지 62에 즉시 반복만을 개선점으로 삼은 피드백이 없다.
- 메시지 64의 교정 이유가 `is`에서 `are`로 바뀐 실제 수 일치를 설명한다.
- 3회 모두 `nativeScore`가 83~87이고 `starRating=2.5`다.
- 세션 총평에 모든 답변이 자연스럽거나 완벽했다는 모순된 칭찬이 없다.
- 최종 구조 실패와 메시지 누락이 0건이다.
- 각 단계의 호출 시간과 세션 전체 시간을 기록해 2회 정상 경로의 실제 지연을 확인한다.

품질 보고서에는 fixture SHA-256, 모델명, 실행 시각, 실행 횟수와 각 결과를 저장한다. 시크릿과 raw provider 응답 메타데이터는 저장하지 않는다.

## 예상 효과와 남는 위험

첫 번째 호출의 의미 오판을 두 번째 호출이 바로잡을 수 있으면서 정상 호출 수는 2회로 유지된다. 의미 판정은 프롬프트와 모델 재심사에 집중되고 서버는 결정적 계약만 검사하므로 LAN-175 사례별 하드코딩이 새로운 질문에 누적되지 않는다.

남는 위험은 두 모델 호출이 같은 오판을 반복하거나 2차 실패 시 1차 오판이 fallback으로 사용되는 경우다. 이를 숨기지 않고 실제 세션 3회 재생에서 후보와 최종 결과, 복구율, fallback률과 지연을 함께 기록한다. 품질 기준을 만족하지 못하면 3차 호출을 추가하지 않고 프롬프트의 일반 판정 기준과 대표 대비 예시를 수정한 뒤 같은 fixture로 다시 검증한다.
